# Schema Lifecycle Management

**Date:** 2026-05-04
**Status:** Design
**Problem:** The TypeDB schema is purely additive — skills add types but can never remove them. 5 dead namespaces (111 types, 0 instances) bloat the schema. No dependency tracking, no retirement path, no namespace documentation.

**Process per namespace** (order matters):
1. Delete any relation instances with that prefix
2. `undefine` relation types (removes roles, unblocks entity removal)
3. `undefine` entity types (leaves → roots)
4. `undefine` orphaned attribute types unique to that namespace
5. Verify: query for any remaining types with that prefix

**After cleanup:** Back up the clean database via `make db-export`.

**What stays:** core (26 types, 150K instances), scilit (25, 29K), dm (33, 28K), skilllog (5, 6K), jobhunt (30, 777), tech-recon (5, 725), trend (2, 48), agentic-memory extensions.

## Part 2: Namespace Declaration in skill.yaml

Each skill declares its schema metadata in `skill.yaml`:

```yaml
name: tech-recon
# ... existing fields ...

schema:
  namespace: tech-recon          # prefix for all types this skill owns
  depends_on:                    # skills whose schemas must load first
    - scientific-literature      # because we add plays to scilit-paper
```

**Rules:**
- `namespace` is the hyphen-delimited prefix that ALL types owned by this skill start with (e.g., `jobhunt` means `jobhunt-position`, `jobhunt-company`, etc.)
- Skills that only extend core types (like agentic-memory, which adds `operator-user` to `person`) use `namespace: null` or omit it
- `depends_on` lists skill **names** (not namespace prefixes) that must have their schema loaded first
- A skill without `schema.tql` doesn't need a schema section

**Examples:**

```yaml
# skills/tech-recon/skill.yaml
schema:
  namespace: tech-recon
  depends_on:
    - scientific-literature

# local_skills/jobhunt/skill.yaml
schema:
  namespace: jobhunt
  depends_on: []

# skills/agentic-memory/skill.yaml
schema:
  namespace: null  # extends core types, no namespace prefix
  depends_on: []
```

## Part 3: Compile into skills-registry.yaml

`make build-skills` already resolves the registry. After resolution, it reads each resolved skill's `skill.yaml` and compiles a `schema_map` section into `skills-registry.yaml`:

```yaml
# Auto-generated by make build-skills — do not edit manually
schema_map:
  load_order:
    - agentic-memory       # no deps
    - scientific-literature # no deps
    - jobhunt              # no deps
    - literature-trends    # no deps
    - curation-skill-builder # no deps
    - tech-recon           # depends_on: scientific-literature
  namespaces:
    jobhunt:
      skill: jobhunt
      schema: local_skills/jobhunt/schema.tql
      depends_on: []
    tech-recon:
      skill: tech-recon
      schema: skills/tech-recon/schema.tql
      depends_on: [scientific-literature]
    scilit:
      skill: scientific-literature
      schema: local_skills/scientific-literature/schema.tql
      depends_on: []
```

## Part 4: db-init Uses Topological Sort

`scripts/db_init.py` is modified to:

1. Accept `--registry skills-registry.yaml` flag (in addition to raw schema paths)
2. When registry is provided, read `schema_map.load_order` to determine order
3. Load `alhazen_notebook.tql` first (always — the base)
4. Load skill schemas in `load_order` sequence
5. Load `namespaces/*.tql` last (infrastructure)
6. Fail with clear error if a dependency is missing or there's a cycle

**Backward compatible:** The raw positional-args mode still works for manual schema loading.

## Part 5: Retire Skill Command

New Makefile target:

```bash
make db-retire-skill SKILL=they-said-whaaa
```

Implementation (new script `scripts/db_retire_skill.py`):

1. Read the skill's namespace from compiled `skills-registry.yaml`
2. Connect to TypeDB
3. For each relation type with that prefix: delete all instances, then undefine the type
4. For each entity type with that prefix: delete all instances, then undefine the type
5. For each attribute type with that prefix: check if orphaned (no remaining owner), then undefine
6. Verify: zero types remaining with that prefix
7. Print summary: types removed, instances deleted

**Safety:**
- `--dry-run` flag: shows what would be removed without executing
- Refuses to retire namespaces with cross-skill dependents (e.g., can't retire `scilit` if `tech-recon` depends on it)
- Backs up before removal (calls `make db-export` first)

## Part 6: Schema Audit Command

New Makefile target:

```bash
make db-audit
```

Implementation (new script `scripts/db_audit.py`):

1. Query running TypeDB for all types
2. Group by namespace prefix
3. Count instances per namespace
4. Flag namespaces with 0 instances as "empty"
5. Cross-reference with `skills-registry.yaml` to identify:
   - Schema types with no owning skill (orphaned)
   - Skills registered but with no schema loaded
   - Dependency violations (type references an unloaded namespace)
6. Print table: namespace, skill, types, instances, status

## Part 7: Namespace Rules in Curation-Skill-Builder

The curation-skill-builder guides the design of new TypeDB-backed skills. Currently, Phase 2 (`add-entity-schema`) records type descriptions as free text but does **not** validate namespace prefixes, generate `schema.tql`, or write the `skill.yaml` schema section. The `_template/` has a `DOMAIN-*` placeholder convention in comments but no enforcement.

### Namespace Naming Convention

Formalized as rules the curation-skill-builder enforces:

1. **Every skill with a `schema.tql` must declare a namespace prefix** in `skill.yaml` under `schema.namespace`
2. **The prefix is a single lowercase hyphenated token** (e.g., `jobhunt`, `tech-recon`, `scilit`). No dots, underscores, or camelCase.
3. **All entity, relation, and attribute types owned by the skill must start with the namespace prefix** followed by a hyphen (e.g., `jobhunt-position`, `jobhunt-company`). Exception: skills that only extend core types (e.g., agentic-memory adds `operator-user` to `person`) declare `namespace: null`.
4. **Do not reuse another skill's namespace prefix.** The compiled registry enforces uniqueness.
5. **Cross-namespace references require `depends_on`** — if your `schema.tql` adds a `plays` clause to a type from another namespace (e.g., `entity scilit-paper plays system-paper-reference:referenced-paper`), you must declare that skill in `depends_on`.
6. **Core types (from `alhazen_notebook.tql`) are always available** — no `depends_on` needed for `domain-thing`, `collection`, `note`, `artifact`, `fragment`, `person`, etc.

### Template Updates (`skills/_template/`)

**`skill.yaml`** — add schema section to the template:

```yaml
# Schema namespace — all types in schema.tql must start with this prefix
schema:
  namespace: DOMAIN              # e.g., jobhunt, tech-recon, scilit
  depends_on: []                 # skill names whose schemas must load first
```

**`schema.tql`** — update header from "TypeDB 2.x" to "TypeDB 3.x" and add namespace validation comment:

```
# Namespace prefix: DOMAIN-*
# ALL entity, relation, and attribute types below MUST start with "DOMAIN-"
# This prefix is declared in skill.yaml → schema.namespace
```

### Curation-Skill-Builder Enforcement

**Phase 2 (`add-entity-schema`)** gains validation:

1. When recording entity schema descriptions, the skill-builder checks that type names in the description match the domain's declared namespace prefix
2. New command: `validate-namespace --skill-dir <path>` — parses `schema.tql` and `skill.yaml`, verifies:
   - All types start with the declared namespace prefix (or are `plays`/`owns` on core types)
   - `depends_on` covers all cross-namespace type references
   - No duplicate namespace prefixes across the registry
3. `make build-skills` calls `validate-namespace` for each skill during resolution. Prints warnings for violations (not hard failures, to avoid blocking builds during development).

### SKILL.md Update

The curation-skill-builder's SKILL.md gains a "Namespace Rules" section documenting the 6 rules above, referenced during the Phase 2 workflow.
